# Project Context

Этот файл нужен как короткая память проекта для нового чата с Codex.

## Что делаем

Проект: Telegram userbot для поиска горячих проектных заявок в чатах и каналах.

Главная цель: ловить сообщения, где человек ищет исполнителя под конкретную задачу:

- сделать/доработать сайт, лендинг, бот, интеграцию, CRM, парсер;
- сделать дизайн, инфографику, карточки маркетплейсов, обложки;
- настроить рекламу, SMM, SEO, маркетинговую аналитику;
- помочь селлеру WB/Ozon/marketplaces с карточками, аудитом, операционкой.

Не нужны:

- вакансии в штат/команду, оклад, full-time, "без опыта", "всему обучим";
- самопиар исполнителей, резюме, портфолио, "помогу", "беру проекты";
- Kwork/биржи/боты, где надо покупать контакты;
- чаты с постоянным мусором, саморекламой и почти без реальных заявок;
- обсуждения без запроса на исполнителя.

Репозиторий: `https://github.com/Ggosh1/tg-bot`

## Где смотреть отчеты

Основные файлы отчетов:

- `lidohod/data/reports/lead_events.tsv` - все решения парсера.
- `lidohod/data/reports/accepted_leads.tsv` - принятые лиды.
- `lidohod/data/reports/rejected_leads.tsv` - отклоненные сообщения.
- `lidohod/data/reports/lead_stats_daily.tsv` - дневная статистика.
- `lidohod/data/app.db` - SQLite база с сохраненными лидами.

Ключевые поля:

- `decision`: `ACCEPTED` или `REJECTED`.
- `reason`: причина решения.
- `ai_class`: `BUYER_LEAD`, `JOB`, `DISCUSSION`, `SELF_PROMO`.
- `subcategory`: `development`, `design`, `marketing_smm`, `marketplace_ops`, `other`.

Для дневных отчетов считать день по Москве. Пример окна для 26.04.2026 МСК:

- UTC start: `2026-04-25T21:00:00Z`
- UTC end: `2026-04-26T21:00:00Z`

- 9 явных мусорных сообщений были ошибочно accepted.

Приоритетные фиксы:

1. Не терять лиды при `FRONTEND_DELIVERY_ERROR`: retry/queue/local pending.
2. Исправить битые ссылки на пользователя и источник.
3. Ужесточить мусор: спамеры, USDT, самопиар, обсуждения без запроса.
4. Отдельно резать вакансии в найм, даже если текст содержит "нужен дизайнер/менеджер".

## Как проверять новые ниши

Не верить гипотезе "там будет много заявок" без замера.

Для каждой ниши нужно собрать 50-100 кандидатов чатов/каналов и прогнать аудит последних 7-14 дней.

Ниша считается перспективной, если после ручной чистки:

- минимум 0.2-0.3 чистых проектных лида на чат в день;
- на 100 чатов получается 20-30+ чистых проектных лидов в день;
- доля мусора и самопиара не съедает поток;
- есть понятный оффер, который можно быстро продавать.

Что считать по нишам:

- `lead_like_per_day`: все похожие на заявки сообщения.
- `clean_project_leads_per_day`: заявки после ручной чистки.
- `jobs_per_day`: вакансии/найм, не считать как проектные лиды.
- `self_promo_per_day`: самопиар, мусор.
- `contact_quality`: есть ли нормальный контакт, а не бот-гейт.
- `duplicate_rate`: повторы одного объявления.

Сначала проверять не количеством аккаунтов, а качеством источников.
Много аккаунтов Telegram может дать баны, flood wait, дубли и операционный шум.

## Ниши-кандидаты

Гипотезы, которые нужно замерить цифрами:

- `marketplaces`: WB/Ozon селлеры, карточки, инфографика, SEO карточек, аудит.
- `realtors`: сайты объектов/агентств, CRM, лидогенерация, реклама, квизы.
- `beauty`: запись, SMM, автоворонки, лендинги, реклама, боты для салонов.
- `legal`: сайты юристов, лидогенерация, реклама, CRM, боты для консультаций.
- `medical`: запись, сайты клиник, реклама, CRM, аналитика.
- `local_business`: общепит, услуги, школы, офлайн-бизнес.

Текущий поток лучше всего идет через `design + marketplaces`.
Чистая разработка идет слабее, поэтому оффер по dev лучше упаковывать прикладно:

- "правки сайта за 24 часа";
- "Tilda/Bitrix/GetCourse/amoCRM интеграции";
- "бот для заявок";
- "автоматизация заявок и таблиц";
- "лендинг + форма + уведомления + CRM".
